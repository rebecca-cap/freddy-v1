OSP proto-base + permanent prototype independence

Single source of truth for this work — written to survive a /clear.
Researched & verified live: prod repo, proto-base, project-hub, OSP knowledge docs (multiple dynamic workflows + a live proof on :5173).

TL;DR Two goals, one core fix

Goal 1 — A separate proto-base for the OSP (Online Selling Platform / "Selling Platform" / "Buy Now"), selectable in the Project Hub alongside the Pricing Engine.

Goal 2 — Permanent independence: any prototype or proto-base must keep working no matter what changes in another. (The grizmo :5181 break exposed that they're currently coupled.)

The key realization: both goals are the same fix — stop sharing node_modules. Every prototype and proto-base should own its dependencies, frozen to its own lockfile. The OSP becomes its own clone; existing prototypes get their own installed deps; the Hub stops symlinking. After that, a dependency change in one place can never break another.

1 What the OSP is

From the Gravitate_OSP_AI_Knowledge docs + the live prod code.

A B2B fuel e-commerce system. Buyers (counterparties) log in, see a filtered grid of fuel products at terminals/racks, enter gallon orders against real-time available volume, and get a TABS-generated loading number on confirmation. Sellers manage volume allocations; price marketers build index-based formulas that drive the live display prices. Hard rules: never oversell inventory, lock the price at order time, only show products the buyer's relationship allows.

browses products, places gallon orders, gets loading numbers
Order
created → validated → confirmed → allocated → loaded
Allocation
available volume pool; decrements per order
Product × Location
one grid row = a fuel SKU at a terminal/rack
Market indices
OPIS/Platts/NYMEX feeds driving formulas

5 The independence problem (root cause of the grizmo break)

The Hub symlinks every prototype's node_modules to the one proto-base node_modules (project-hub/server/node-modules.ts → ensureNodeModulesSymlink, called at scaffold + before each run). It's a speed/disk shortcut, but it creates two coupling points:

  • Shared dependency versions — every prototype runs on the proto-base's current versions, ignoring its own lockfile.
  • Shared Vite cache (node_modules/.vite/deps) — one dev server's re-optimization can corrupt another's.

What happened: grizmo's own lockfile pins react-router 6.16 (has unstable_useBlocker); the symlink fed it the proto-base's June-bumped 6.28.1 (removed it). grizmo coasted on a stale Vite cache until a second dev server rebuilt that shared cache and exposed the mismatch. test-3 + test-1 carry the same latent issue. The symlink is the coupling; the version bump was the landmine.

6 Recommended architecture

The principle Every prototype and every proto-base owns its own dependencies, frozen to its own committed lockfile. Nothing shared at runtime.
  1. Retire the node_modules symlink. Replace it with a real per-project install (yarn install --immutable from each project's own lockfile). One install at scaffold; complete isolation thereafter. This alone makes the grizmo class of break impossible — a base bump can't reach an existing prototype, and separate node_modules = separate Vite caches.
  2. OSP = a separate proto-base clone (e.g. ~/freddy/proto-base-osp) with its own deps + lockfile. PE and OSP then evolve fully independently; syncing OSP from prod can never touch PE or its prototypes.
  3. Guardrail: a "smoke-test all prototypes" Hub command that boots each one and confirms it renders — run after any base change so regressions surface immediately.

Disk / speed trade-off (why the symlink existed)

Per-project installs cost ~1–2 GB and a few minutes each. Options, ranked:

OptionIndependenceCost
Per-project yarn installfull~1–2 GB each, zero toolchain riskrecommended now
pnpm (shared content store)fulltiny disk, small migrationif many prototypes
APFS copy-on-write cloneisolation onlyinstant/cheap, but freezes at base's current versionsnew projects only
Recommendation: plain per-project installs now (no toolchain risk, proven). Consider pnpm later if the prototype count grows.

7 Hub changes

The Hub already manages one proto-base as a start/stoppable runtime (__proto-base__, :5173) and already has a disabled "Selling Platform" sidebar stub. Adding the OSP is small because the runner is generic:

  • server/node-modules.ts — ensureNodeModulesSymlink → ensureNodeModulesInstalled (real yarn install --immutable). Callers: dev-server.ts:90, projects.ts:51.
  • server/paths.ts + server/dev-server.ts — generalize the single proto-base into a small registry (pe, ss, later sdi); add SELLING_PLATFORM_PATH + a new port (e.g. 5175).
  • server/index.ts + src/lib/api.ts — parameterize the /api/proto-base/:id/... routes + client calls by id.
  • src/components/Sidebar.tsx — flip the disabled "Selling Platform" stub into a live ProtoBaseControl (its product dot is already ss).
  • New: a smoke-test script/command that boots every prototype + proto-base and checks it renders.

A registry (vs copy-pasting a second set) makes the third product, S&D, a one-line entry later.
