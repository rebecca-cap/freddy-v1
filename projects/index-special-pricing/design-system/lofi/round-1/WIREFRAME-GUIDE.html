Wireframe Index & Guide · Round 1

Index Special Offers — Quick Create

A fresh, true lo-fi exploration of a single-view Quick Create for an index special offer in the Gravitate selling platform. The goal set in the kickoff: let a sales rep fire off an index offer in roughly 10–12 clicks and under 30 seconds — down from ~25 clicks and a minute today — without cramming the flow into the old quick-create drawer.

Project
Index Special Offers (OSP)
Round
1 · kickoff (Reece + Agustin, Jun 15 sync)
Designer
Frank Overland
Date
June 2026

Overview

These five wireframes are self-contained, static HTML prototypes built from the vendored Gravitate WireframeDesignSystem (wf- components, --wf- tokens). They cover one job: a power-user sales rep creating and sending an index-priced special offer on a Group 3 gas product. Index offers are the case where today's quick-create drawer breaks down, because index pricing needs choices the drawer never represented.

This is a direction-finding round, not a requirements build. Per the kickoff, current designs are "more a feedback canvas than ready requirements," so round 1 deliberately starts from a clean slate to test whether a from-scratch single-view flow lands somewhere better than the iterated drawer.

The premise

"If I'm a sales rep sitting at lunch and the market moves and I want to fire off an offer fast — what we have today is too cumbersome." The whole round optimizes for that moment.

The user flow

The screens follow the speed path end-to-end: Launch → (prefill) → one-view fill → Send → payoff.

Shared layout structure

The two quick-create screens (02, 03) deliberately reuse the production "Create New Offer" surface so they read as the same page the wizard-based offers already use — stakeholders shouldn't have to learn a new screen. That surface is a full-screen takeover with no app sidebar:

  • Top app bar (lf-topbar) — a × close affordance and the bold "Create New Offer" title, matching the real create flow. A small dashed "Round 1 index" link sits at the right as reviewer-only scaffolding.
  • "Back to Offers" link (lf-backtolist) above the content, as on the production page.
  • Two-column body (lf-offer-grid) — a wide left content card (wf-card) holding the whole offer, and a narrower (~360px) right "What customers will see" preview card (lf-preview) with the same four labelled sections the wizard shows: Product & Location, Pricing Strategy (Index Pricing), Timing Windows, Target Customers.
  • Footer action bar (lf-offer-footer) — occupies the wizard footer's position, but since this is a single view it carries Cancel + Send offer with a muted speed note instead of Previous / Step N of 4 / Next.

Screens 01 (offers list) and 05 (sent confirmation) still use the app shell (wf-page-with-sidebar) since they are not the create-offer surface; the mobile screen (04) uses its own phone frame.

Fit the existing surface, drop the stepper

Per Frank's R1 feedback, quick-create adopts the real Create New Offer chrome (top bar, Back to Offers, left content + right "what customers will see") so it feels native alongside the wizard-based offers. The only structural change is removing the stepper: the wizard's gated steps collapse into one scannable card, so the rep tweaks the few cells that matter (mainly the price formula) and sends. Inside the card, groups are plain heading-plus-fields rather than wf-form-section blocks, to respect the "no form-section inside a card" rule.

Color & status language

Color follows the design system's reservation rule — status colors mean status, primary blue means interactive intent, everything else is neutral. The specific signals used in this round:

  • Primary blue — the Send CTA, the selected duration/group chip (.lf-chip.is-selected), the left edge of the price-formula readout, and the key preview section's left edge. Interactive intent and the user's current choices only.
  • Success green — reserved for genuine status: a wf-badge-success "Live" offer and the green metric cards on the Offer sent scorecard.
  • Warning amber — a wf-badge-warning "Closing" offer in the list. Pending/attention only.
  • Error red — only on a real validation gap: the unconfirmed Volume field on screen 03 (wf-field-error + inline message + the rail showing "needs input").
  • Formula readout — the index price (.lf-readout) gets a primary left border to mark it as the screen's most important decision, with the result rendered in wf-mono.
  • Designer annotations (.lf-anno, dashed border) — lo-fi commentary for reviewers, not product UI.

Per-screen documentation

01-launchpad.html

— Offers launchpad Open wireframe →

Screen purpose

Where speed starts. The rep lives on the Offers page; the fast paths must be one click away, so the primary + Create button (shortcut ⌘N) opens a wf-menu shown here in its open state.

Component breakdown

The Create menu lists three items: Quick create — Index special (⌘N, → screen 02), Create from prior (⌘P, → screen 03), and, below a divider, Full setup (prescriptive) — the existing detailed flow, kept as the no-template fallback. Below an annotation banner, a wf-action-bar ("Recent offers · 12 active", product filter, search, Export) sits over a wf-datagrid of four recent offers.

Data shown

Each grid row shows the product/terminal, the index price formula in wf-mono (e.g. OPIS G3 +$0.0250), volume, the recipient group, and a status badge — Live, Closing, and neutral "Filled".

02-quick-create-index.html

— Quick create (the hero) Open wireframe →

Screen purpose

The core proposal: author a complete index offer from one view, inside the standard Create New Offer page (top bar + "Back to Offers", left content card, right "What customers will see" preview). No app sidebar, no stepper — the wizard's steps collapse into one card.

Left content card — five groups (heading + fields)

A · Start — a Template select (Group 3 Suboctane — OPIS, etc., plus "No template (start blank)") beside a Create from prior button (⌘P). Either prefills everything below.
B · Product & volume — a 3-up grid: Product, Terminal/location, Volume (42,000 gal).
C · Price formula (the meat) — Index (OPIS/Platts/Argus Group 3), Index product (Suboctane Gross/Net…), and a Differential in wf-mono (+0.0250). Below it the price readout (.lf-readout) renders the resulting price — OPIS Group 3 Suboctane Gross + $0.0250/gal — and the note "Settles & invoices at lift time."
D · Timing — durations, not dates. Two chip groups: Visible for (2h / 4h / Until close / Custom) and Pickup window (24h / 48h / 3 days / Custom).
E · Invite — existing groups as selectable chips with member counts: Gulf Coast (3) selected, Midcon (5), Chicago (4), Rack — Tulsa (6), + Add customer. A helper line summarizes "3 customers selected · Gulf Coast group."
Groups are separated by hairlines (lf-group) rather than wrapped in wf-form-section cards, since the whole thing already sits in one content card.

Right preview — "What customers will see"

The same four labelled sections the wizard preview uses: Product & Location, Pricing Strategy → Index Pricing (formula in wf-mono, "Invoices at lift time", marked as the key section with a primary left edge), Timing Windows (Visible 4h · Pickup 48h), Target Customers (Gulf Coast group · 3 customers). Unlike the wizard's "Not configured" placeholders, these are filled because template/prior prefills the offer.

Footer

A footer action bar in the wizard's footer position: Cancel (back to Offers), a muted speed note ("target ≤ 12 clicks · ≤ 30s"), and the primary Send offer (⌘↵). No "Step N of 4" since there are no steps.

Centerpiece

The price formula is the only field that requires real thought; the readout makes "index + differential, invoiced at lift" legible at a glance. Everything else is a pick from presets.

03-create-from-prior.html

— Create from prior Open wireframe →

Screen purpose

Abdul's create-from-prior pattern, but intersecting the quick-create view instead of a separate wizard — and on the same Create New Offer surface as screen 02. The rep lands on a near-complete offer cloned from a previous send.

Initial state

Inside the content card, a wf-alert-info banner: "Prefilled from a prior offer — cloned from Suboctane · Tulsa · sent May 28" (with a "Change source" action). Prefilled fields carry an italic kept from prior tag (product, terminal, index, index product, timing, invite). Two fields draw the eye: Volume is empty and in a wf-field-error state ("Confirm volume for this offer"), and Differential shows a helper "Prior was +$0.0225 · market moved."

Right preview

The same "What customers will see" panel, but the Product & Location section flags Volume — needs input — making it obvious that prefill collapses the work to the two cells that actually changed. The footer note reads "From prior · ~3 clicks to send."

The intersection

Create-from-prior reuses the exact Create New Offer surface. One mental model, two entry speeds.

04-mobile-quick-create.html

— Mobile quick create Open wireframe →

Screen purpose

Tests Reece's ask that the field sequence also work on mobile. Shown in a 390px phone frame.

Layout

A compact app bar (back, "Index special / Group 3 gas", Draft badge) over a single-column body that keeps the exact desktop order — Start → Product & volume → Price formula → Timing → Invite — with wf-dividers between zones and the same .lf-readout for the formula. A pinned send bar at the bottom carries the tap-count meter ("8 of 12 taps, ~26s") and a full-width Send button.

05-review-send.html

— Offer sent Open wireframe →

Screen purpose

The payoff. A wf-toast-success confirms "Index special sent to 3 customers," and the page shows the created offer plus a speed scorecard.

Component breakdown

A wf-card for "Suboctane · Tulsa" (ref OFR-2026-0617-14, wf-badge-success "Live") with wf-data-rows for the full price formula, "Invoices: At lift time," volume, visibility window, pickup, and recipients; footer actions Save as template / Duplicate / Withdraw. Beside it, two wf-metric-card-success cards — 9 clicks (target ≤ 12) and 24s (target ≤ 30s) — and an annotation contrasting that with today's ~25 clicks / ~1 min.

Design decisions reflected

  1. One scrollless view, not a wizard. Index quick-create is a single surface (no wf-stepper); the rep scans and tweaks rather than paging through gated steps. Directly answers "don't make it a cousin in a drawer."
  2. The price formula is the centerpiece. Index + differential gets its own emphasized section and a live readout that states it settles/invoices at lift time — the one field that needs real thought.
  3. Template and create-from-prior share the same surface. Both prefill the quick-create view; create-from-prior is the same screen at a higher starting speed, not a separate wizard.
  4. Timing is expressed as durations, not dates. "Visible for" and "Pickup window" use duration chips that roll from "now," matching the rep's mental model and cutting date-picker clicks.
  5. Invite by existing groups. Group chips with member counts (Gulf Coast = 3) let one tap add a whole audience, reusing the platform's groups rather than picking customers one by one.
  6. The speed budget stays visible. A muted speed note in the footer (and the post-send scorecard on screen 05) keep the ≤12-clicks / ≤30s goal testable, without cluttering the customer-facing preview. (An earlier draft put a live click meter in the right rail; that was dropped when the rail became the production "What customers will see" preview.)
  7. Keyboard-first power use. Shortcuts (⌘N, ⌘P, ⌘↵, ⌘F, G) are surfaced on-screen; chips are focusable. The flow never requires a mouse.
  8. No AI in this design. Per the kickoff, speed comes from templates, prior-cloning, durations, and groups — not assistance.
  9. The prescriptive flow stays as a fallback. "Full setup" remains in the Create menu for the no-template case; this round designs the template/prior happy path, not the deep authoring flow.
  10. Built strictly from the design system. Only real wf- classes and --wf- tokens; lo-fi scaffolding is isolated under a separate lf- namespace so it never leaks into a production read.

Open for round 2

  1. Essential clicks: which fields are truly the "10–12 essential clicks" vs. deferrable to advanced/collapsed? The hero currently shows everything expanded.
  2. Desktop vs mobile: one shared flow or a segmented mobile design? Does the right-rail live preview need a mobile equivalent (a peek sheet), or is the pinned tap-count enough?
  3. No-template branch: how prescriptive must the fallback be, and how is the hand-off from "blank" to full setup surfaced inside the single view?
  4. Duration controls: do "Visible for / Pickup window" need custom ranges beyond presets, and how is "Until close" defined against market hours?
  5. Differential entry: is a plain ± input enough, or do reps need step/nudge controls and a market reference (e.g. "prior was +$0.0225") inline?
  6. Requirements confidence: are business rules defined enough to commit, or does round 1 stay a direction-finding canvas? (Reece/Agustin flagged current designs as "more a feedback canvas.")

How to view

Open any file in wireframes/ directly in a browser — each is self-contained static HTML, no build step or server. Start from index.html for the gallery.

Recommended sequence:

  1. 01-launchpad.html — see the Create menu and its three paths; note the price formula already reads in the recent-offers grid.
  2. 02-quick-create-index.html — the hero. Read top-to-bottom; the price-formula readout and the right-rail click meter are the things to react to.
  3. 03-create-from-prior.html — same view prefilled; note the kept from prior tags and that only Volume + Differential need attention (meter drops to 3 of 12).
  4. 04-mobile-quick-create.html — confirm the field order survives the phone form factor.
  5. 05-review-send.html — the payoff and the 9-clicks / 24s scorecard.
