Wireframe Index & Guide · Round 3

Index Special Offers — Two-section Create

Round 3 reworks round 2's one-page Create New Offer into a two-step flow — Step 1 · Offer details and Step 2 · Time frame. The preset starts blank with two save-back paths (Update / Save as new, the name disclosed progressively), the offer-type dropdown is gone (everything is Index Special), the differential moves below the variable grid, customer chips are readable and visible across every invite tab, and timing is rebuilt to mirror the proto-base-osp component: Visibility and Pickup windows with absolute start/end date·time·timezone, real-clock dropdowns (15-min) whose quick-starts do the math, and a Calendar View that visualizes both windows. Reflects the second review pass on Jun 23; the screens are interactive end-to-end.

Overview

Five static-but-interactive HTML wireframes continuing the Index Special Offers quick-create flow, built from the vendored Gravitate WireframeDesignSystem (wf- components, --wf- tokens; lo-fi scaffolding isolated under lf-). Round 2 settled the real price-formula model and a progressive invite; the round-2 review accepted both and turned to form shape and field interactions — the page had gotten long as fields accumulated, the preset wrongly assumed a default, and timing couldn't express a future start.

The review in one line

"We'll go from four to two… and then if we can find ways to make it shorter then maybe it gets moved in and it's one page." Make the interactions with the fields really good, don't strip the fields.

Changes from Round 2

Round 3 ran through two review passes. The first pass split the page into Sections A/B; the second pass (Jun 23) refined that into a true two-step stepper and rebuilt timing against the production prototype. The list below reflects the current, shipped state.

  • Two-step stepper (edit 1). Product/formula, timing and invite were four stacked groups. R3 splits the flow into Step 1 · Offer details (product + price formula + customers) and Step 2 · Time frame (timing), with a step indicator and Previous / Next · "Step N of 2" footer — mirroring the production create wizard. See screen 02.
  • No offer-type dropdown. Everything here is an Index Special deal, so the offer-type selector was removed; Step 1 opens straight into Product / Terminal / Volume.
  • Preset starts blank, with two save paths (edit 2). Default = no preset, everything empty. Selecting a preset populates every field; editing a populated preset reveals just two paths — Update preset and Save as new preset. The "you changed this" label and the "continue as one-off" path were dropped; choosing Save as new progressively discloses the new-preset name field.
  • Differential moved below the grid; display label fixed (edit 3). Build the formula, then basis (Argus/rack), then the differential last, so it sits beneath the grid. The read-only result is labelled "Price formula (display)" (was "resulting price").
  • Consolidated, tab-agnostic customer chips (edit 4). A running chip bar stacks every selected customer with its source tag and persists across tab switches. Selected chips were re-styled to a white surface with a blue border and dark text (previously blue-fill + black text, unreadable). See screen 04.
  • Timing rebuilt to match the prototype + Calendar View (edits 5, 6). Replaces the R2 duration model with the proto-base-osp SelectTimingWindows pattern: a Visibility Window and a Pickup Window, each with a Start Now checkbox and absolute Start / End rows (date at time + America/Chicago tag). Time pickers list real clock times at 15-minute intervals; the quick-starts (Now / +1h / +6h / 6 PM / Midnight) live in that dropdown and compute the clock time for you. A Calendar View below visualizes both windows (Visibility blue, Pickup green, overlap split), supports month nav, and "click any date to fill the next empty field." Inline validation enforces Pickup start ≥ Visibility start. The offset case is expressed naturally by giving each window its own start.
  • Files renamed for the new shape. 02-quick-create-index → 02-create-offer, 04-invite-modes → 04-invite, 05-review-send → 05-offer-sent. Launchpad (01) and price-formula chooser (03) carry forward largely unchanged.

Shared layout structure

Carried forward from round 2. The create surface (02, 03, 04) uses the production Create New Offer takeover — no app sidebar:

  • Top app bar (lf-topbar) — × close + bold "Create New Offer" title; a dashed "Round 3 index" reviewer link at the right.
  • "Back to Offers" link (lf-backtolist) above the content.
  • Two-step stepper (lf-stepper + lf-steppanel) on the hero — a numbered indicator (1 Offer details · 2 Time frame) with clickable steps; one panel shows at a time.
  • Wizard footer (lf-wizfoot) — Previous · "Step N of 2" · Next (Next becomes Send offer on step 2). Full-width single column, matching the production create wizard (no preview rail this round).
  • Bottom-sheet template chooser — the Formula Template Selector (lf-bottomsheet, ≈85vh) over a dimmed page for the "choose formula" state.

Screens 01 (offers list) and 05 (sent) keep the app shell (wf-page-with-sidebar) since they aren't the create surface.

Color & status language

Unchanged in intent from round 2 — status colors mean status, primary blue means interactive intent / current selection / current location, everything else neutral.

  • Primary blue — Send CTA, selected chip / tab (wf-tab-active), the active step circle, and the border on selected customer chips. Note the calendar uses its own pair — Visibility blue (#dbeafe fill / #4386f6 edge) and Pickup green (#dcfce7 / #32ae5e) — carried verbatim from the prototype to keep parity.
  • Success green — real status only: the "Populated from preset" / "Preset updated" badge and the per-window completion check.
  • Warning amber — the "Edited · unsaved" preset badge, a stretched credit utilization (86%).
  • Error red — validation: "Pickup start must be at or after Visibility start."
  • Monospace (wf-mono) — formula variable cells, differential, volume, the price-formula display string, credit figures.

Per-screen documentation

01-launchpad.html

— Offers launchpadOpen wireframe →

Screen purpose

The offers list and entry point, in the app shell, carried forward from round 2. The "+ New offer" menu lists New — Fast / New — Full, then Start from a Preset and Create from prior. The "New — Fast" path now opens 02-create-offer.html.

02-create-offer.html

— Create New Offer, two-step (hero)Open wireframe →

Screen purpose

The whole offer as a two-step flow. This is where edits 1–6 land. The screen is interactive: a preset dropdown that populates/clears, a stepper, customer chips that stack across tabs, real-clock time pickers, and a calendar that fills and visualizes the windows.

Preset row + initial state (edit 2)

On load no preset is selected and everything is blank: the price-formula shows its empty state, the customer chip bar reads "Nobody selected yet," and a helper line states "Nothing is assumed." Choosing a preset populates the formula grid and shows a green "Populated from preset" badge; editing any field then reveals two save paths — Update preset and Save as new preset — with an amber "Edited · unsaved" badge. Clicking Save as new preset progressively discloses the new-preset name field.

Step 1 · Offer details (edits 3, 4)

Product: Product, Terminal/location, Volume — no offer-type dropdown (all Index Special).
Price formula: a chosen-formula line + the editable variable grid (%, Publisher, Instrument, Type, Date Rule, Display, row options); then Basis (Argus/rack) and the Differential on their own row below the grid (the last formula decision); then the read-only "Price formula (display)" string.
Customers: the consolidated chip bar on top, then the four-tab picker (Groups / Authorized / Credit / All). Selecting from any tab adds a white, blue-bordered chip with a source tag; chips persist across tab switches; the × removes the chip and unchecks its row everywhere it appears.

Step 2 · Time frame (edits 5, 6)

Mirrors the production SelectTimingWindows. Two window cards — Visibility Window ("when customers can see and bid on your deal") and Pickup Window ("when customers can pickup their product") — each with a Start Now checkbox and Start / End rows of date at time + an America/Chicago tag. Checking Start Now greys the start fields and sets them to "Now." The time field opens a dropdown of real clock times at 15-minute intervals, with quick-starts (Now / +1h / +6h / 6 PM / Midnight) pinned on top that compute the clock time. Below, a Calendar View visualizes both windows (Visibility blue, Pickup green, overlap a diagonal split), navigates by month, and "clicking any date fills the next empty field" in sequence (Visibility start → end → Pickup start → end → reset). Inline red text enforces Pickup start must be at or after Visibility start.

Offset, expressed naturally

Because each window owns its own start, the offset case (take orders today, lift next week) needs no special mode — set Visibility Start Now and pick a later Pickup start. The calendar shows the gap between the two ranges directly.

Still open

The preset save-path button treatment, and the calendar's fat-finger "go back" recovery, aren't final — the interactions are what's being validated here.

04-invite.html

— Customers: consolidated pickerOpen wireframe →

Screen purpose

The full-room version of Side A's customer step, demonstrating edit 4 at scale. The mode tab changes what you're picking from; the running selection stays visible regardless of tab.

The consolidated chip bar

A sticky chip bar pins below the header. Each pick — a favorite group chip, or a checked row in any tab — adds a lf-selchip with a source tag (Group / Authorized / Credit / All) and a remove control. Switching tabs swaps the table beneath but leaves the chip list intact, so picks made under "Authorized" and "Credit" are visible together without bouncing between tabs. The same customer appearing in two tabs stays in sync (check one, the other reflects it).

Reece's hardest requirement

"I need to see the output selection of people agnostic of tabs." The chip bar is the single source of truth for who's on the offer; tabs are just routes to find them. Group CRUD is intentionally out — groups are managed in an external group manager; this screen only consumes and selects them.

05-offer-sent.html

— Offer sentOpen wireframe →

Screen purpose

The payoff, carried forward from round 2. A success toast, the created offer's detail card (real formula name + differential, invoices at lift, the invite method used), and a "Reuse this" panel — Save as Preset (primary) above Create from prior, keeping the deliberate-vs-recall distinction concrete.

Design decisions reflected

  1. A two-step flow, not four groups or a long page. Step 1 "Offer details," Step 2 "Time frame." The single page was right mentally but too heavy; two steps keep each surface focused and match the production create wizard's shape.
  2. No offer-type choice. Every deal here is an Index Special, so the selector is removed — Step 1 starts at the product.
  3. Nothing is assumed — the preset starts blank. Default selects no preset and leaves every field empty; you opt into a preset rather than having one presumed for you.
  4. Two preset save paths, disclosed progressively. Editing a populated preset offers just Update preset or Save as new preset — no "continue one-off," no "you changed this" preamble; the new-preset name appears only after you choose Save as new. "Update" is the verb for the resave path.
  5. The differential is the last formula decision. Build the variable grid, choose Argus vs rack basis, then set the differential — so it sits below the grid, mentally subordinate to the formula it adjusts.
  6. The result field is a display, not an input. Labelled "Price formula (display)" and read-only — it reflects the grid + differential; it's the string customers see, not a place to author.
  7. Selected customers are tab-agnostic, and the chips are readable. One running chip list is the source of truth for who's invited; chips use a white surface + blue border + dark text so they read clearly (the blue-fill + black text was illegible).
  8. Timing matches the production prototype. Visibility and Pickup windows with absolute start/end date·time·timezone, real-clock pickers at 15-min intervals, and quick-starts that compute the clock time — adopting proto-base-osp's SelectTimingWindows rather than inventing a duration model.
  9. A calendar visualizes the windows. The two windows are easy to mis-read as numbers; the calendar shows Visibility (blue) and Pickup (green) ranges, their overlap, and fills the next empty field on click — fast once learned, and it makes the offset gap visible.
  10. Two windows, each with its own start, expresses the offset case for free. Visibility = how long you're taking orders; Pickup = the date-time range to lift once confirmed. Decoupled starts cover live-now, scheduled, and offset without a separate mode.

Open for round 4

  1. Preset save-path buttons: final labels and arrangement for Update / Save as new (interaction validated; treatment still open).
  2. Calendar fat-finger recovery: the calendar is kept (fast once learned) but there's no clean way to go back after a mis-tap; the current click sequence resets after all four dates are set — confirm that's the recovery, or add an explicit undo.
  3. Quick-starts set: confirm the in-dropdown quick-starts (Now / +1h / +6h / 6 PM / Midnight) and whether midnight should be the default start.
  4. Step count: two steps now; if Step 1 gets heavy do we split product/formula from customers into a third step, or hold at two?
  5. Hi-fi + customer testing: timing-related customer testing is on hold until this is in hi-fi (customers are still on the original version).

How to view

Open any file in wireframes/ directly in a browser — self-contained static HTML with in-page JavaScript, no build step. Start from index.html.

  1. 02-create-offer.html — the hero. It loads empty; pick a preset to populate, then edit any field to reveal the two save paths (and click Save as new preset to see the name appear). Note the differential below the grid in Step 1.
  2. In Step 1's Customers block, pick from different tabs and watch the chips stack and persist; remove one with its ×.
  3. Click Next (or the step indicator) to Step 2. Open a time field for the quick-starts + 15-min clock list; tick Start Now to grey a start; then click dates in the calendar to fill Visibility start → end → Pickup start → end and watch the blue/green ranges render. Try a Pickup start before the Visibility start to trip the validation.
  4. 04-invite.html — the same consolidated chips with more room; pick across all four tabs.
  5. 01 / 03 / 05 — entry, formula chooser, and payoff, carried forward.
