The one-paragraph version

Executive summary

The real story isn't slowness, it's lack of a finish line. The actual feature — the two-step preset-led Quick-Create Special Offer drawer — was assembled correctly and verified live on day 1 (R2, Jun24). Everything after that, roughly 20 of the ~22 rounds, was visual polish and rework, and that long tail is where the pain lived: a cyan-vs-navy color decision that flipped three times, the same excalibrr gap={N} no-op gotcha rediscovered in at least five separate sessions, AG-grid and antd-v4 traps re-hit despite being in memory, and several regressions (copy-button outside the box, preset Save dirty-flag, spacing collapse) that shipped because side-effects weren't checked before declaring done. Claude's craft was genuinely strong inside each round — disciplined Playwright verification, real reuse instead of rebuilds, careful scoping that protected the full-setup wizard — but it had no mechanism to say "this is good enough, stop." Two compounding risks: the whole feature is STILL uncommitted/unpushed past R5, so weeks of work sit in a working tree, and the favorites bug Frank reported was never reproduced or root-caused before the thread ended. The fix is process, not capability: a gotchas pre-flight, batched polish passes with a round budget, confirm-then-build on ambiguous asks, a regression check before "done," and a commit cadence.

Credit where due

What went right

The craft inside each round was genuinely strong. These are real strengths worth keeping.

Core feature built fast and correctly on day 1

R2 (Jun24-b) correctly reframed the job as assembly/reuse rather than a from-scratch rebuild — Claude discovered the prototype already contained the real production Special Offers feature and wired its existing pickers into the approved two-step flow. End-to-end paths (preset population, dirty/save, validation, Step 2, Send, handoff) were Playwright-verified the same day.

Evidence
  • 'the prototype already contains the real production Special Offers feature ... The hi-fi job is to assemble these into the approved two-step preset-led quick-create flow, not build from scratch.'
  • 'Verification caught a real bug: the preset's TotalVolume (42,000) is less than the template's max-per-order (50,000), so the volume validator rejects and blocks Next.'
  • Round log: R2 (Jun24) CORE FEATURE built and verified day 1

Disciplined live verification was the norm, not the exception

Across nearly every session, changes were confirmed in a running browser with Playwright — computed-style checks, pixel measurement, console inspection — instead of being asserted from the diff. This caught genuine defects and avoided phantom-bug reports.

Evidence
  • 'Select panel = 8px, Select input box = 5px'; 'minute column shows only 00/15/30/45'
  • 'the grid toolbar says 4 Selected while the bin shows 2 ... The live toolbar actually reads 2 Selected — the screenshot just caught a mid-update frame. All counts agree'
  • 1153 thinking blocks and heavy Read/Bash verification effort across 11 sessions

Reuse-first instincts instead of rebuilding

Repeatedly chose to wire existing components with back-compatible props rather than build parallel copies — the shared template selector got an initialFilters prop, the full-wizard PreviewPanel was reused for the live rail, volumeInputList and toggleGroup were reused for overrides and chips.

Evidence
  • 'add the back-compatible initialFilters prop to the shared template selector' (after Frank warned 'It looks like you were trying to rebuild it')
  • 'Reused the existing PreviewPanel (the same tile component the full 4-step wizard uses) rather than rebuild'
  • 'displayGroups ... plus toggleGroup is already in scope ... net -18 lines'

Scoping that protected the existing full-setup wizard

Shared-component edits were consistently made opt-in (new optional props defaulting to current behavior) and gated behind variant='quick', then regression-spot-checked against the full 4-step wizard so it never broke.

Evidence
  • 'make chrome-strip opt-in via new optional props (defaults preserve current behavior) — zero risk to the full wizard'
  • 'timing restructure gated behind variant=quick (full wizard untouched)'
  • 'full-setup primary button stays navy rgb(15,17,33) ... header stays light #EEF0F8 after each color pass'

Honest flagging of limitations and contradictions

Claude surfaced spec/implementation mismatches and instruction contradictions to Frank instead of silently guessing or silently flip-flopping, and admitted its own regressions plainly.

Evidence
  • 'Reece describes the customer filters as additive toggles ... but in the prototype they're mutually-exclusive tabs -> raised via AskUserQuestion before building'
  • 'There's also a button-color instruction that contradicts the earlier navy, no cyan decision — I'll need to clarify that'
  • 'the going backwards feeling is fair. That copy-button regression was mine: I shipped a fragile positioning fix and verified it by looking at a screenshot instead of actually measuring'

Clean commit hygiene when commits did happen

When Claude committed, it staged only src/, deliberately excluding package.json/yarn.lock churn and .playwright-cli/scratch PNG junk per the workspace memory rules.

Evidence
  • '18 source files staged; package.json, yarn.lock, and .playwright-cli/ correctly excluded'
  • 'Staged only src/, excluding package.json file: path dirt, yarn.lock, .playwright-cli/ and scratch PNGs; committed d062547'
The honest part

What went wrong

Root-caused and sorted by severity. Most of these are process gaps, not capability gaps.

No 'good enough' line — endless polish micro-iteration

high

Root cause. Neither Claude nor the workflow had a defined finish state or round/effort budget for the polish phase, so each Frank observation spawned another full open-browser-edit-measure cycle. Polish was handled reactively, one instance at a time, instead of batched.

The core feature was done in R2 but ~20 more rounds of spacing, color, rounding, and copy tweaks followed. Single-file or single-number changes routinely triggered multi-agent analysis and long verification loops. The metrics show the scale: 4073 assistant turns and 1818 tool calls (813 Bash) across the week, the vast majority post-feature.

Evidence
  • Round log: ~22 rounds, core done R2, the rest polish/rework
  • Frank: 'i'm really kind of surprised the context window is so high. Are you using efficient ways to do this with loops?'
  • Jun30-b polish session: 489 asst turns / 242 tool calls for a UI pass; Jun30-c: 53 Bash calls for a handful of margin changes

Same framework gotchas rediscovered over and over

high

Root cause. Documented gotchas were not loaded/checked before writing the kind of code that trips them. Memory existed but wasn't consulted pre-flight; it was written to after each rediscovery instead of read before.

The excalibrr Vertical/Horizontal gap={N} no-op, the !important utility-class override fight, the antd-v4 theme.useToken() crash, GraviButton-ignores-children, and AG-grid quirks were each hit multiple times — despite several already being documented in MEMORY.md and gotchas.md. The gap no-op alone surfaced as a 'discovery' in at least five sessions (R2c-fix, Jun25-a, Jun25-b, Jun30-b, Jun30-c).

Evidence
  • 'The catalog already has a Vertical/Horizontal entry (line 12).' — yet the regression occurred and was only found after Frank flagged it
  • 'Every touching gap issue ... traces to the same root cause: excalibrr Horizontal/Vertical ignore the gap={N} prop (8 instances in the timing files alone)'
  • 'GraviButton ignores children (known gotcha) — I must use buttonText'; 'MEMORY already lists this exact [useToken] gotcha'

Regressions shipped because side-effects weren't checked before 'done'

high

Root cause. Verification confirmed the intended change rendered but did not check for collateral breakage; screenshot-glancing substituted for measurement on positioning fixes.

Several defects were introduced and declared complete, then caught by Frank: the copy button rendered outside the form field, the spacing collapse from swapping margins for no-op gaps, and the preset Save/Update dirty-flag that stopped firing. The copy-button case is the clearest — it was 'verified' by glancing at a screenshot rather than measuring.

Evidence
  • Frank: 'now the copy icon is outside of the form field. What is going on? Why are you making such weird design decisions?'
  • Frank: 'There seems to be no gap between the items in each section ... everything is touching.'
  • Frank: 'the save as new preset or update preset feature that we used to have is no longer working'

Large uncommitted/unpushed pile — bus-factor and loss risk

high

Root cause. No commit cadence during the polish phase; commits happened opportunistically rather than at the end of each verified round.

The feature is still uncommitted past R5 and not pushed. Weeks of polish across many sessions sit in a working tree on project/index-special-pricing, recoverable only via memory resume docs. A crash or accidental /clear already cost real recovery time once.

Evidence
  • Brief: 'Feature is STILL UNCOMMITTED past R5 / not pushed'
  • Jun26 session opened by reconstructing a crashed session's state after an accidental /clear
  • Multiple memory docs note work 'UNCOMMITTED' as the resume state

Color decision churned three times (cyan <-> navy)

medium

Root cause. Shared between Claude and Frank — Frank changed his mind, but the decisions were also never locked in a single durable place, so each session re-litigated them. Largely attributable to Frank's direction, but the process offered no decision log to anchor against.

The primary accent was decided and fully re-implemented three times across R3b/R3c/R3d and Jun30-b: black/white -> cyan -> navy 'no cyan' -> cyan CTAs on navy chrome. Each swing meant rewriting the same CSS color blocks, and work from one round was thrown away and redone twice.

Evidence
  • #8 'Footer Next button: cyan'; #9 'drop cyan -> navy blue ... zero cyan'; #10 'make the primary buttons ... cyan ... but make sure the cyan buttons have white text'
  • Round log: R3b CYAN -> R3c NAVY (reverted R3b) -> R3d CYAN CTAs (flips again)
  • '#9 most of it is removing my cyan overrides'; '#10 Recolor the in-drawer primary buttons to cyan + white (re-adding what #9 removed)'

Intent misreads from building before confirming

medium

Root cause. Acted on an interpretation of an ambiguous instruction instead of confirming scope first; misread Reece's note and Frank's deictic references.

On ambiguous asks, Claude sometimes built the wrong thing and had to rip it out: the Offer/Auction deal-type dimension Frank never wanted (whole sub-round to remove), the R10 'tight spacing' read as internal rows when Frank meant flanking space, and cutting the All-customers tab without checking the remaining tabs had data.

Evidence
  • Frank: 'Okay, in the toggles, there is no offer or auction, it's only index and fixed.' — full sub-round to remove the dimension
  • Frank: 'no the space between the formula differential and the Price formula display' (R10 misread)
  • Frank: 'i need some authorized to lift to choose from we would have a group if it had no customers'

Unresolved user-reported bug never root-caused

medium

Root cause. When the test environment couldn't reproduce a user-reported failure, Claude concluded 'works on my end' rather than investigating the environment gap (stale build, cache, different state path).

Frank reported the favorites star not adding to favorites; Claude's repeated live tests all passed, so the discrepancy between Frank's real environment and Claude's passing tests was never reconciled before the thread ended.

Evidence
  • Frank: 'I see the star in the row, but it's not adding anything to the favorites'
  • Claude: 'I just re-verified it live on a freshly reloaded page ... So the wiring works on a clean load.' — turn ends with discrepancy unexplained

The point of all this

What to improve

Seven concrete changes to how we work. None require new capability — they're guardrails that give the work a finish line.

1

Gotchas pre-flight before writing UI code

Do this. Before any spacing/styling/grid edit in this proto-base, read gotchas.md and the relevant MEMORY gotcha files and state the applicable traps up front (gap={N} is a no-op -> use style gap or mb-*; excalibrr utilities ship !important; GraviButton needs buttonText; antd-v4 useToken shim; AG-grid first-mount zero-column race). Treat memory as a checklist to read, not just a place to write after getting burned.

Why. The same five-ish gotchas were rediscovered across at least 5 sessions despite being documented. Reading them costs seconds; rediscovering each costs a regression and a Frank correction.

Eliminates most of the repeated-gotcha class (counted ~13 times across sessions) and the regressions that flow from them.

2

Batch polish into one scoped pass with a round budget

Do this. When entering a polish phase, do a single full design sweep (spacing, color, typography, rounding) against the target wireframe, list every issue, fix them in one pass, then verify once. Set an explicit budget (e.g. 2 polish rounds) and surface to Frank when it's exceeded: 'we're past budget — is this shippable or is there a specific blocker?'

Why. Polish was handled one reactive instance at a time, so Frank re-reported the same class of issue (gaps, spacing) round after round and the session count ballooned to ~22.

Collapses the ~20-round tail toward 3-5 rounds; gives the work a finish line.

3

Confirm-then-build on any ambiguous or deictic ask

Do this. When an instruction is ambiguous (a 'this/that' reference, a third-party note like Reece's, a 'make it tighter' with no anchor), restate the interpretation in one line and either get a yes or check the data/screenshot before building. Reserve building-first for unambiguous asks.

Why. Every intent misread (Offer/Auction, R10 flanking spacing, empty All-customers tab) came from acting on a guess. A one-line confirm would have prevented the discarded sub-rounds.

Removes the misread-driven dead-ends (~4 events), each of which cost a partial or full round.

4

Regression check before declaring done

Do this. After any visual/positioning fix, verify the specific thing changed AND spot-check the neighbors: measure (don't glance at a screenshot) that the element is still inside its container, re-check adjacent spacing, and re-run the one workflow most likely to break. For positioning, assert bounding-box containment, not visual plausibility.

Why. The copy-button-outside-the-box and spacing-collapse regressions both passed a screenshot glance and shipped. Claude itself diagnosed the root cause as 'verified by looking at a screenshot instead of actually measuring.'

Catches the regression class (~7 shipped regressions) before Frank does.

5

Commit at the end of every verified round

Do this. After each round passes verification, commit src/ with a one-line round summary (per existing hygiene rules). Push at least daily. Stop treating commit as a separate end-of-everything event.

Why. The feature is uncommitted past R5 and a crash already forced a costly recovery. A week of polish living in one working tree is pure downside risk.

Eliminates loss/recovery risk and makes /clear genuinely safe at any point.

6

Lock decisions in a durable decision log

Do this. Maintain a short DECISIONS section (in the project memory or PROJECT_NOTES) with the current locked answer to recurring questions — accent color, button style, title scheme. On any reversal, update the one entry rather than re-deriving from scratch each session.

Why. The cyan/navy color flipped three times partly because there was no single anchored decision to point back to; each session re-litigated it.

Stops decision re-litigation churn; makes flip-flops a one-line edit instead of a CSS rewrite.

7

Treat 'can't reproduce a user-reported bug' as an environment investigation, not a pass

Do this. When Frank reports a failure that tests pass, do not conclude 'works on my end.' Investigate the gap: confirm Frank's build/branch, check for stale HMR/cache, ask what exact steps and which environment, and diff the state path Frank uses vs the test path.

Why. The favorites bug was left unresolved because passing tests were treated as the final word over Frank's lived failure.

Closes the loop on real defects instead of leaving the user with a known-broken feeling.

For transparency

How this retro was made

Source. 11 Claude Code session transcripts from Jun 24–30, 2026 (~40 MB of JSONL), plus the project memory round-log and the prototype's git history.

Process. Each transcript was compressed to a high-signal digest (Frank's verbatim messages, assistant summaries, per-turn tool tallies). A workflow then fanned out one analyst agent per session to extract wins, friction, and quotes, and a synthesis agent consolidated them into the themes, timeline, and ledger you see here. 12 agents ran in total; the heavy transcript reading happened in their context, not in the main thread, to keep the working context small.

One caveat on the numbers. The raw totals (4073 assistant turns, 1818 tool calls, 813 Bash, 401 Edits, ~349 Frank screenshots, 1153 thinking blocks) are large for one drawer feature, and the shape confirms the thesis: effort was back-loaded into polish, not the build. R2 produced the working feature in one session; the remaining ten sessions are dominated by verification round-trips (Bash 813, Reads 387) on small CSS/UX tweaks. The 31 subagent spawns and frequent multi-agent recon for what resolved to one-to-five-file edits show analysis overhead disproportionate to change size. Do not read the duration_min column as work time — sessions left open overnight (e.g. 1143, 1006, 1070 min) inflate it; the honest effort proxy is 117 Frank review messages and ~22 rounds, both heavily weighted toward polish and rework rather than new capability.
