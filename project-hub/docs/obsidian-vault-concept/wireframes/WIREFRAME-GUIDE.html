Concept Wireframes · Exploration

Bringing the Obsidian vault into Project Hub

Six low-fi screens that pitch one idea: move Frank's product-design Obsidian vault into the shared Project Hub repo, and bridge it with the Hub so the team inherits the reasoning behind the work — not just the prototypes. These are rough wireframes for a conversation, not a build spec. The rationale is backed by research; sources are listed at the end.

Overview

Frank keeps two things. The Project Hub is a React + Express app that manages design prototype projects through their life — lo-fi, hi-fi, handoff — and renders the design system as live reference pages. The Obsidian vault is a personal knowledge base: folders for Decisions, Patterns, Learnings, Projects, Retros, and Syncs, with per-type templates and [[wikilinks]] that tie the notes into a graph. The Hub is shared on GitHub. The vault is not — it lives on one machine.

These hold the same knowledge in two modes. The Hub is operational: what exists and whether it runs. The vault is cognitive: why it was built that way and what was learned. This set of wireframes proposes co-locating the vault in the repo so it ships with the Hub, then connecting the two so each project shows its own decisions and learnings, and so teammates who never open Obsidian can still read the second brain in the browser.

Each screen is a self-contained HTML file. They share one stylesheet, wf-base.css, built from the Hub's WireframeDesignSystem tokens, so the wireframes match the look of the app they describe. No build step, no server, no JavaScript.

The two surfaces

One rule runs through every screen: split the work by how fast it changes, not by topic. Markdown is great for durable, written knowledge and bad as a design-system spec, because it can't sync token or component values from code and silently goes stale. So the vault owns the durable narrative and the Hub owns the generated, live specs.

The vault owns (durable, narrative)The Hub owns (generated, live)
Decisions — why we chose X, what we rejectedToken & component values, from the JSON source
Patterns — reusable how-to conventionsInteractive specimens (/design-system, /wireframe-system)
Learnings — gotchas, root cause, how to avoidRuntime — which prototypes run, on which port
Projects, Retros, Syncs — the story and the graphProject lifecycle — phase, versions, covers, handoff
Design-system narrative — principles, do/don't—

The bridge between them is one frontmatter field. Every note can carry project: "[[Grizmo]]". That single link is the join key: the Hub reads it to group a project's notes, and the note links back to the project. One field, both directions.

Color language

The wireframes use color to mark which surface you're looking at.

  • Obsidian purple (#7c3aed) — the vault: wikilink pills, "Open in Obsidian" buttons, vault nodes, vault-mechanics callouts.
  • Hub blue (#2563eb) — the app: primary actions, project nodes, the active KB nav item.
  • Hub green (#6AC08B) — the Project Hub brand mark in the sidebar.
  • accepted / shipped   proposed / high-severity   today's gap.

Callouts come in three flavors: blue for general notes, purple for how the vault mechanics work, amber for the "why this matters" reasoning.

01-concept-overview.html

— Two surfaces, one brainOpen wireframe →

What it shows

The thesis screen. It sets up the whole pitch so the other five have context. Read this first.

On the screen

  • Architecture diagram — three boxes left to right: Project Hub (operational), the shared GitHub repo in the middle (where both the app and the vault live), and the Obsidian vault (cognitive), with two-way arrows. Below it, the join-key note: project: [[Quote Filter Bar]].
  • Who owns what — two cards splitting durable/narrative (vault) from generated/live (Hub), with the line that vault DS notes link to the Hub specs instead of restating values.
  • What changes — a today (siloed, red) vs after (shared, green) comparison.
  • Six moves — a table of the proposed moves with a rough value and lift read, and the screen number that wireframes each.

Why it leads

The volatility split is the one idea most likely to be argued, so it goes up front. It also answers the obvious objection — "isn't this just docs?" — by showing the vault and Hub doing different jobs.

02-repo-and-toggle.html

— Co-locate the vaultOpen wireframe →

What it shows

Move 1: where the vault physically goes, and how a teammate gets to it. The cheapest move and the precondition for all the others.

On the screen

  • Repo tree — the project-hub repo with a new top-level vault/ next to server/, src/, resources/, docs/. The vault holds the existing folders (Decisions, Patterns, Learnings, Projects, Retros, Syncs, Design System, Templates), a .obsidian/ config, and a README.md.
  • The gitignore — an allowlist: ignore .obsidian/*, then re-include the shareable config (app.json, appearance.json, community-plugins.json, core-plugins.json, hotkeys.json, snippets/). Always ignore workspace.json, workspace-mobile.json, cache, .trash/, .DS_Store.
  • Vault switcher — a mock of Obsidian's "Open vault" panel listing "Frank's Vault" and an active "Project Hub Vault," plus "Open folder as vault…". A hotkey toggles between them.
  • Onboarding — a three-step list for the repo README: clone, open the folder as a vault, toggle.

The one gotcha

The Obsidian Git plugin needs the vault root to be the git root — one repo, one vault. Here the vault is a subfolder, so that plugin won't drive it. The screen says to skip the plugin and let the repo's normal commit and push be the sync path, which is the flow the team already uses for the Hub code. The gitignore matters because workspace.json is per-person pane layout and would conflict on nearly every pull.

03-kb-browser.html

— The KB stub goes liveOpen wireframe →

What it shows

Move 2, and the highest-value one. The Hub sidebar already has an "Obsidian KB" item that's a disabled stub. This makes it a real page: a read-only vault browser inside the Hub.

On the screen

  • Hub chrome — the real sidebar (Project Hub mark, Projects, and the System section: Design System, Design Patterns, Wireframe DS, and Obsidian KB now active).
  • Three-pane browser — a folder rail with live counts (Decisions 6, Patterns 9, Learnings 7, Projects 5, Design System 15, Retros 1, Syncs 2, Templates 7); a note list for the open folder; and the rendered note.
  • Rendered note — "Tailwind Merge Class Conflicts" with its frontmatter shown (type, tags, severity: high, project: "[[GDS Design System]]"), the Problem / Root Cause / Fix headings, and a Related section with two wikilink pills.
  • Open in Obsidian — a purple button that fires an obsidian:// deep link for anyone who wants the graph or wants to edit.

How it works

Because the vault is markdown files in the same repo, the Express side reads them directly. It globs vault/, parses each note with gray-matter into frontmatter and body, and serves JSON. A markdown renderer with a wikilink plugin turns [[links]] into in-Hub /kb/:slug anchors. No Obsidian install and no running app are needed to read. This reuses the exact pattern the Hub already uses for its generated design-system reference pages.

Why it matters

Engineers and PMs read the second brain in a browser tab. The vault stops being a tool only one person opens.

05-capture-loop.html

— The vault fills itselfOpen wireframe →

What it shows

Move 4: how the vault stays current without anyone remembering to write notes.

On the screen

  • Three-lane flow — Triggers (Hub events, the PD-sync skill, Claude Code) → Write path (scaffold from templates, patch frontmatter with gray-matter, seed a note) → the Vault → back into the Hub's KB and Project → Knowledge pages.
  • Worked example — Grizmo moves to hi-fi: Frank promotes it in the Hub; the Hub patches Projects/Grizmo.md to status: hifi and leaves the body alone; the KB and project pages reflect it right away.

The rule

Pick one write path. Write to files on disk when Obsidian may be closed; use the Local REST API only when it's guaranteed open. Never both at once, or Obsidian's in-memory copy overwrites the disk edit. The point of the loop is to beat the main failure mode of any knowledge base — notes nobody maintains — by making capture a side effect of work that already happens.

06-graph-and-bases.html

— Graph & ADRsOpen wireframe →

What it shows

Moves 5 and 6, the ones that compound. The graph turns scattered notes into a map of how the product got here; ADR discipline makes that map trustworthy.

On the screen

  • Portfolio graph — an illustrative node graph (inline SVG) where two project nodes, Grizmo and Quote Filter Bar, both link to a shared "GDS Design System" node. The shared node is the point: the graph reveals reuse across projects you didn't plan.
  • Bases dashboard — an "Open Decisions" table (Decision / Status / Project / Date) embedded in an index note with ![[Decisions.base#Open]]. Bases is a native core plugin, so the table renders for every teammate with no install. Dataview stays the power-user fallback.
  • ADR skeleton — a Decision note's frontmatter (type, status, supersedes, deciders) and the MADR headings: Context, Decision Drivers, Considered Options, Decision, Consequences, Related.

The discipline

One decision per file, numbered, immutable once accepted. To change a decision you write a new note that supersedes the old one; you don't edit history. Decisions get reviewed in a PR like code. That audit trail is what makes the log trustworthy, and it's the direct answer to the bus-factor risk of one designer holding all the context.

Design decisions reflected

  1. Split by volatility, not topic. The vault owns durable narrative (decisions, patterns, learnings). The Hub's generated JSON stays the source of truth for token and component values. Vault design-system notes link to the Hub specs rather than restate them, so they can't drift.
  2. One join key. A note's project: "[[..]]" frontmatter is the only link the Hub needs to connect a note to a project. It works both directions and needs no database.
  3. Read without Obsidian. Every bridge feature works off the markdown files on disk, so a teammate who never installs Obsidian still reads the whole knowledge base in the Hub. Deep links to Obsidian are an upgrade, not a requirement.
  4. Sync through the repo, not the plugin. The vault is a subfolder, so the Obsidian Git plugin can't drive it. The repo's normal commit and push is the sync path — one mechanism the team already runs.
  5. Ship a curated config. An allowlist gitignore sends teammates one consistent theme and plugin set while ignoring per-person workspace state. The shared vault should depend on as few plugins as possible.
  6. Capture as a side effect. Notes are created and updated by Hub events, the PD-sync skill, and Claude Code, so the vault stays current without a discipline nobody keeps.
  7. Decisions are immutable ADRs. Numbered, one per file, superseded rather than edited, reviewed in PRs. This is the format generic tools handle worst and the strongest reason to keep design knowledge as markdown in the repo.

Research basis

The recommendations come from a fact-checked scan of how product teams actually use Obsidian, keep vaults in Git, and integrate it with other tools. The load-bearing findings:

  • Obsidian is free for work since Feb 20, 2025 — the commercial license is optional and adds no features. No licensing blocker.
  • Bases is a native core plugin (shipped v1.9, 2025). .base files are plain text, version-control cleanly, and render with no install — the right default over Dataview for a non-developer team.
  • Obsidian Git needs vault-root == git-root. A vault in a subfolder can't use the plugin; the supported escape hatches are a submodule or plain repo git. Confirmed against the plugin docs and forum.
  • Gitignore workspace.json (and other per-user state); commit the shared config via an allowlist. This is the documented fix for constant merge conflicts.
  • ADRs / docs-as-code (MADR, Diátaxis) — keep decision records in the repo, numbered and immutable, organized by reader need. The Decisions folder is already an ADR system in spirit.
  • Markdown loses as a design-system spec — no Figma/code sync means token values drift ("Stale Documentation Syndrome"), which is why the vault references the Hub's generated specs.
  • Integration seams — the obsidian:// URI scheme (plus the Advanced URI plugin for rename-stable links), reading notes on disk with gray-matter, and rendering wikilinks with markdown-it for an in-Hub read-only view.

Open questions

  1. Vault location and name. vault/ at the repo root, or docs/vault/? The vault name is hard-coded in every obsidian:// deep link, so it needs to be stable and documented in the README. Pick one before any link is persisted.
  2. How much to move. The current vault has test notes and design-system stubs mixed with real content. Move the whole thing, or curate first? A shared vault rots faster than a personal one, so a clean starting set may be worth the effort.
  3. Hub render depth. Does the in-Hub KB page need full graph navigation and search, or is folder + note + wikilinks enough for v1? The design-refs browser pattern already covers most of it.
  4. Write-back scope. Which Hub events should patch the vault automatically — just phase changes, or new-project scaffolding and decision seeding too? More automation means more value and more chances to write a note someone didn't want.
  5. Design-system narrative overlap. The Hub already has /design-system, /design-patterns, and /wireframe-system reference pages. Where does the vault's design-system narrative stop and the generated reference begin, so they don't duplicate?
  6. Teammate adoption. Will engineers actually open a second vault, or only ever read the Hub KB page? If it's read-only-in-Hub for everyone but Frank, that's fine — but it changes which features matter.
