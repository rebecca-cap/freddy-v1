Project List — what's real, what's theater

An item-by-item review of the Project List surface (the Hub home page): every control traced to its handler and the data behind it, with a verdict on whether we can make it real on the current setup (Express + JSON-file store + git worktrees + React) and what that would take.

Scope HomePage · ProjectControlBar · ProjectList (cards) · ProjectListView (list) + atoms Verified by reading source + the /api/projects shape Date 2026-06-09
Headline: the data on each card/row is real (name, branch, port, variant count, navigation). What's not real is almost the entire control bar — search and every filter chip are explicitly disabled "coming soon" stubs — plus three card details that present as fact but aren't: the creator (hardcoded "Frank"), the product (a regex guess), and "Updated" (it's actually the created time). Most of it is cheaply makeable; two things genuinely aren't worth it.

The control bar (mostly theater)

Search box

ProjectControlBar.tsx:128 — SearchPlaceholder()
Placeholder✓ Can make real — easiest win on the pageXS
What it is
A <div> styled like an input, aria-disabled, cursor-not-allowed, title "Search — coming soon". It is not an <input> and captures nothing.
Why not real
Never built — placeholder pending a decision on client vs server search.
To make real
HomePage already holds the full projects array in state. Add a search string + filter before passing to the lists — no backend needed.
  1. Make it a real <input> with controlled state in ProjectControlBar (lift to HomePage).
  2. Filter on name + branch (case-insensitive substring) before <ProjectList /> / <ProjectListView />.
  3. Reflect the filtered count in the "All (N)" chip.
Note: PROJECT-DETAIL-PLAN.md:136 already specs a richer scoped search (rounds, commits, resources) as a "P3 server-side ripgrep" — but the basic name/branch filter is pure client-side and ships today.

Status filter chips — Active / Review / Merged

ProjectControlBar.tsx:22–45
Placeholder◑ Partly — "Active" yes; "Review"/"Merged" need a status modelS–M
What it is
"All (N)" is the only active chip — and even it doesn't filter, it just shows the count. Pinned, Active, Review, Merged are all disabled "coming soon".
The catch
Status today is binary: util.statusFor() returns "active" if the dev server is running, else "draft". The chip set promises four states (Active / Review / Merged) that the data can't currently produce — so two of them would always come back empty.
To make real
  • All / Active — wire immediately; filter the in-memory list on derived status. Doable now.
  • Review — no source of truth. Needs an explicit status field on ProjectRecord someone sets — a product decision, not just code.
  • Merged — semi-derivable: promote machinery exists (promotedTo on variants, promoteVariant()). Could mark a project "merged" once promoted, but the rule needs defining.

Product filter + the product chip on every card

filter: ProjectControlBar.tsx:36 · chip: ProductChip.tsx + util.productFor()
PlaceholderGuessed✓ Can make realS
What it is
The "Product: All ▾" dropdown is disabled. The colored product label on every card (Pricing Engine / Selling Solutions / S&D·Incab / Cross-product) looks like real metadata but is a regex guess from the project name + branch (util.productFor). The code itself admits it: "Until backend tags projects per product, derive a product hint…"
Risk
It presents a confident, color-coded fact that's actually inferred — a project named "test-3" silently becomes "Cross-product". That's the kind of "not-real" that misleads rather than just sitting inert.
To make real
  1. Add product to ProjectRecord (server/store.ts).
  2. Capture it in NewProjectDialog (a 4-option select) at create time.
  3. ProductChip reads the real field; keep productFor() as the default/fallback for legacy projects.
  4. Wire the dropdown to filter the list on it.

Creator filter + the avatar & name on every card

filter: ProjectControlBar.tsx:41 · avatar: ProjectList.tsx:60, ProjectListView.tsx:71
PlaceholderHardcoded◑ Technically yes — but low valueS
What it is
Every card hardcodes initials="FO" / "Frank" and every row hardcodes "Frank Overland". There is no creator field in the data. The "Creator: All ▾" filter is disabled.
Honest take
The Hub is a single-user local tool. We can add a creator field and wire the filter, but with one user the avatar is decoration and "Creator: All" filters nothing meaningful. Recommend: don't build the filter — either keep the avatar as static branding or drop the creator column until it's genuinely multi-user.

"Updated <time>"

ProjectList.tsx:71, ProjectListView.tsx:76 → util.relativeTime(p.createdAt)
Mislabeled✓ Can make real — or just relabelXS–S
What it is
The label says "Updated" but the value is createdAt — there is no updatedAt field anywhere. relativeTime() itself is real and correct; it's pointed at the wrong timestamp.
Two options
  • Honest now (XS): relabel to "Created" — zero backend change, instantly truthful.
  • Real "Updated" (S): add updatedAt to ProjectRecord and stamp it on mutations (version stamp, lofi changes, pin, etc.), then point the label at it.

Status chip — two of its four states are unreachable

StatusChip.tsx (defines draft/review/active/merged) · util.statusFor()
Partial◑ See the Status filters above — same root causeM
What it is
StatusChip can render four styled states, but statusFor() only ever returns active (running) or draft (stopped). "In review" and "Merged" are dead code paths today.
To make real
Same fix as the Review/Merged filters: introduce a real status model — an explicit field for "review", a promote-derived rule for "merged". Until then the chip is honest for what it shows, just under-using its vocabulary.

Faked on purpose (call it, don't necessarily fix it)

Card cover art

ProjectCardCover.tsx
Decorative◑ Possible but costly — and a deliberate choiceL
What it is
Four hand-drawn stock SVGs (chart / table / bulk-edit / form) picked by a hash of the slug. Not a screenshot of the project — purely cosmetic variety. The file says so outright.
Can we make it real?
Yes, but it's a real jump. A true preview means screenshotting each prototype via Playwright — which requires the dev server running, is async, and needs a thumbnail cache + invalidation. The stock cover was the intentional dodge of exactly that cost. Recommend leaving it unless previews become a priority; if so, capture a thumbnail on launch/stop and cache it per slug.

Main-prototype "live" badge

HomePage.tsx:81
Hardcoded✓ Can make real — minorS
What it is
The proto-base card always shows a green live badge — it's a literal, not a probe. If :5173 is down, it still says "live".
To make real
A health fetch to :5173 (or a hub-side reachability check) gated behind a small interval. Cheap, slightly flaky across CORS — worth it only if the false-positive ever bites.

If you want to make it real — recommended order

Highest honesty-per-effort first

  1. Relabel "Updated" → "Created" XS — kills a lie in one line, no backend.
  2. Wire the search box XS — client-side name/branch filter; the biggest "why doesn't this work" gap.
  3. Pin / favorites S — clean vertical slice (field + toggle endpoint + filter + sort); good proof the pattern works end-to-end.
  4. Product field + chip + filter S — replaces the misleading regex guess with truth.
  5. Status model M — unlocks Active/Review/Merged filters and the status chip together. Needs a product decision on what "review" means.
  6. Defer: creator filter (single-user → low value), real cover previews (costly), live-probe badge (cosmetic).
The one principle: the inert placeholders (search, pin, filters) are honest — they're visibly disabled and say "coming soon". The three that deserve attention regardless of whether we build the features are the ones that present guesses or stale data as fact: Product (regex guess), Creator (hardcoded), and "Updated" (wrong timestamp). Those mislead; the others merely wait.
