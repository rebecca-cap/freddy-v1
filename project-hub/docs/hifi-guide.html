Hi-fi Management Guide

How versions, variations, and every button on the Project Page's Hi-fi section work. Screenshots are from the live hub (Grizmo project, Growmark variation).

01The mental model

Go right to make a new version of the mainline. Go down to make a per-client variation of a version. Every variation runs on its own localhost.

Mainline [v1]────[v2]────[v3] (+) ← stamp new versions to the right │ ├── [v1b] Client A :5182 ← variations fork downward, └── [v1c] Client B :5183 each on its own port
Versions are save-points, not copies. Only the latest mainline state runs on the project's port. To see an old version live, fork a variation from it.

02The Hi-fi card at a glance

Prototypes tab → Hi-fi. One mainline row of version chips, then one row per variation. No git, no commits — that all lives on the Branch tab.

03Mainline & versions

ElementWhat it does
v1Version chipA named save-point. The latest one is highlighted — that's what the project's dev server serves. Click any chip to fork a variation from that exact version.
＋StampSaves the prototype's current state as the next version (v2, v3, …). Use it when a chunk of work feels "done enough to name."
↗LaunchOpens the mainline prototype in a new tab. Enabled only while the project's dev server is running (start it from the bar above the tabs).

04Variation rows

A variation is a full per-client copy: its own branch, its own folder, its own port. Change the data for a client there — the mainline never sees it unless you promote.

After a variation is promoted, it gets a promoted → v3 badge so you can see its work already lives in the mainline. The variation keeps running for its client.

05The five buttons, left to right

#ButtonWhat it does
1Start / Stop (▶ / ◼)Spins this variation's dev server up or down on its own port. Runs alongside the mainline and other variations — open them all in different tabs.
2Copy CLAUDE.md pathCopies the variation's CLAUDE.md path to your clipboard. Paste it into a Claude Code session and Claude is locked to this variation's worktree — it won't touch the mainline or other clients. The icon flips to a green check when copied.
3Pull from projectMerges the mainline into this variation. This is how a client copy picks up new features while keeping its own data. Conflicts are listed below the row; the variation is left untouched until you resolve them.
4Promote to mainlineThe reverse: merges this variation into the mainline, then automatically stamps the next version. Asks for confirmation first. Use it when work you did in a variation should become the new baseline.
5Delete variationRemoves the variation's branch, folder, and worktree after a confirmation. The mainline and other variations are unaffected. Deleted letters are never reused (delete v1b, the next fork of v1 is v1c).

06Forking a variation

Two ways in: click a version chip (preset to that version) or the New variation button in the card header (preset to the latest version).

The fork captures the version exactly as stamped — even if the mainline has moved on since. That's how you revisit an old version live.

07Branch tab — inspecting under the hood

When you do want to see git (commits, ahead/behind), the Branch tab has it. The dropdown switches between the mainline and any variation. It's inspect-only — nothing gets checked out or moved.

08Guard rails — when the hub says no

Every merge-ish action protects your work. These are the messages you might see and what they mean.

Stamping with unsaved work "There are unsaved changes from your last session. Ask Claude to commit them, then stamp this version." The prototype folder has edits that aren't committed yet. Finish (or ask Claude Code to commit), then stamp.
Stamping with nothing new "Nothing new since v3 — the prototype hasn't changed." The mainline hasn't changed since the last stamp. Nothing to save.
Merge conflicts (Pull or Promote) "Merge conflicts — resolve in the worktree, then refresh:" followed by a file list. Both sides changed the same files. The merge was safely rolled back — nothing is half-merged. Resolve in a Claude Code session in that worktree, then pull/promote again.
Promote succeeded — the mainline gains the variation's work as a merge, the next version is stamped automatically, and the variation shows a promoted → vN badge.

09Recipes

New client copy

  1. Click the version chip you want to copy (or New variation for latest)
  2. Name it after the client → Create variation
  3. ▶ Start its dev server, open its localhost
  4. Copy CLAUDE.md path → paste into Claude Code → change the data there

Ship a feature to every client

  1. Build the feature on the mainline (project worktree)
  2. ＋ Stamp the new version
  3. On each variation row, hit Pull from project
  4. Each client copy keeps its data, gains the feature

Save your progress as a version

  1. Finish a chunk of mainline work (committed)
  2. Hit ＋ on the mainline row
  3. The new chip becomes the highlighted latest version

Client work becomes the new baseline

  1. On the variation row, hit Promote to mainline
  2. Confirm — the merge lands and the next version auto-stamps
  3. Other clients pick it up via Pull from project
