3 · Why a plain “pull” is the wrong move

Merging origin/master into proto-base or any existing project branch drags the antd 4→5 + excalibrr 4→5 migration into prototypes that were styled and built against antd 4. Every in-flight prototype would need component-level rework, and my excalibrr fork's customizations (the -osp line) don't exist in the published 5.x package. For throwaway-speed prototypes, that cost is not worth it.

So: two tracks, split at the version boundary.

4 · The plan

Track A — New projects are born current

  • Hub cuts new worktrees from origin/master instead of the local stale branch (one-line change at the git worktree add start point, server/projects.ts:172).
  • Because master's dep is a registry version, the Hub's excalibrr repoint/symlink becomes a no-op for these projects; yarn install just works (my npm token already authenticates to registry.npmjs.org). Repoint logic (server/node-modules.ts:83-88) becomes conditional: only rewrite when the dep starts with file:.
  • New projects are antd 5 + excalibrr 5 from day one, matching what you ship.

Track B — Existing projects update on demand, selectively

  • Existing antd-4 prototypes stay on the antd-4 base. No forced migration; they age out naturally as projects finish.
  • When one needs an upstream fix (e.g. the mock-auth fix), I git cherry-pick that commit into the project branch rather than merging all of master.
  • The Hub grows a small “Base updates” feature: git fetch + a per-project “N commits behind master” badge + a cherry-pick/merge action. It slots directly beside the existing local variant-sync (server/projects.ts:570-595, mergeInto in server/git.ts:94-127); the only missing primitive is a fetch wrapper in git.ts.
